# 提取仓库的 backlog 治理原则

体检用户指定仓库(未指定则用当前仓库)的 backlog 治理现状,**对照下方固定标尺**逐条判定,只用证据说话,产出结构化报告。若用户提供了上一次的报告路径,则额外输出增量。

## 评判标尺(固定 6 条——勿改,改了历次报告就不可比)

backlog = 「已识别但未完成」工作的有序存储,天生有 6 种失败模式,每条原则对偶一种:

1. **唯一队列**——堵碎片化:待办是否只在一个权威入口?
2. **意图先于执行**——堵做错东西:每项是否写清 做什么/为什么/可判定验收?
3. **原子可逆**——堵范围蔓延:一项是否可独立验收、独立回退?
4. **顺序有据**——堵阻塞/白忙:排序是否反映真实依赖与价值?
5. **完成即证明**——堵假完成:"完成"是否对照 DoD 可验证、优先机器验?
6. **同步现实、陈旧即剪**——堵漂移:是否定期对账、删除过时项?

(可选支撑项:**可追溯**——每项是否链得回来源/决策/变更。)

## 步骤

1. **先跑数据采集脚本拿"数字证据"**(脚本只产出数字,不下判断):
   ```bash
   python3 governance_stats.py --repo <仓库路径> --rules-file <治理文档,如 docs/backlog.md> --limit 50
   ```
   纯标准库无依赖;有 `gh` CLI 则拉最近 N 个已合并 PR(标题/正文/改动文件/评审/CI/关联 issue),否则回退 `git log`。它给出:
   - **structural**:原子性(每 PR 改动文件数)、可追溯(关联 issue 比例)、完成验证(评审/CI 覆盖比例);
   - **rule_hits**:每条编号规则被历史引用的次数与命中率——`dead_rule_candidate=true` 是**死规则/过度设计**信号(`null`=样本不足,勿据此下结论);
   - **principle_signals**:6 条原则各自词汇在过程历史里的出现频次;
   - **governance_files**:仓库内 backlog/治理类文件。
2. **再人读关键文件补语境**(脚本看不到语义):backlog/治理文档全文、`AGENTS.md`/`CLAUDE.md`、变更日志、若干 PR 评审线程。
3. **逐条判定**这 6 条。每条给四件套:**机制**(本仓用什么手段实现或没有)/ **证据**(`file:line` 或 `PR#`,读不到写"未发现")/ **判定**(`健全`/`隐性未成文`/`缺失`)/ **历史失败**(该原则对应的失败在 PR/commit/changelog 里是否真发生过,举证)。
4. **两类横切发现**:**过度设计**(成文规则中找不到对应历史失败、却增加负担的——优先看脚本的 `dead_rule_candidate`)/ **真实缺口**(历史反复发生的失败却无规则覆盖的)。
5. **增量(若给了上次报告)**:输出自上次以来的变化——新增/废除规则、新发生的失败、缺口闭合或新增、判定升降级。

## 输出

一份 markdown 报告(默认 `backlog_governance_review_<日期>.md`,或按用户指定路径),含:概览表(6 原则 × 判定 × 一句话证据)/ 逐条详情 / 过度设计清单 + 真实缺口清单 / 增量 / 一句话健康度结论。

## 纪律(硬约束——违反则报告作废)

- **证据优先**:每个判断引用 `file:line`/`PR#` 或脚本数字;无证据写"未发现",绝不编造。
- **数字归脚本,判断归你**:脚本给频次与命中率,但"原则是否被满足"由你结合语义判定;样本不足(`dead_rule_candidate=null`)时不下死规则结论。
- **区分原则与机制**:不因某机制(依赖 DAG、宪法分层、风险分级…)有名就判"缺失"。单人小项目用线性清单,不算缺依赖图——**判的是原则是否被满足,不是机制像不像大厂**。
- **无发生的失败不立建议**:不主张引入任何规则,除非能指出它对应的、本仓真实发生过(或迫近)的失败。
- **精简**:报告 ≤ 1 页;细节用链接指回源文件。
- **不写回被测仓库的 backlog**:报告落在独立目录或单独仓库,避免"双事实源"。
